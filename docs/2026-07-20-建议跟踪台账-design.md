# 建议跟踪台账 — 设计说明

**日期**：2026-07-20  
**状态**：首版已落地（`reports/建议跟踪台账.md`，2026-07-20 回填全部最终报告）  
**目标**：对所有给出明确买卖建议的标的，同时支持**条件监控**与**复盘命中率**；与 thesis 重点看板分层，避免用深度论文去追全部研究标的。

**关联**：[`docs/2026-07-16-重点标的看板-design.md`](2026-07-16-重点标的看板-design.md)（thesis / 指定标的总览）

---

## 1. 问题与目标

`reports/` 下研究很多，但多数只有终稿建议，没有可追踪的「条件是否达成 / 事后对不对」。

现有 `重点标的看板` + `/thesis-tracker` 只覆盖少数量 thesis，无法覆盖全部带买卖建议的报告。

本设计要回答：

1. 当初建议的价格/事件条件到了没有？（监控）
2. 这条建议事后算命中、未命中，还是条件一直未达？（复盘）
3. 哪些该升格成 thesis，哪些只保留轻量台账行？

---

## 2. 方案选择

采用**方案 2：独立建议登记簿 + 看板摘要**。

| 对比项 | 本方案 | 未采用 |
|--------|--------|--------|
| 主文件 | `reports/建议跟踪台账.md` 活文档 | 塞进重点标的看板主表 |
| 粒度 | 按「建议事件」一行（同公司可多条） | 每公司一个 call-card 文件（方案 3，后置） |
| 深度 | 轻量字段；升格才建 thesis | 全部建完整 thesis |
| 自动化 | 首版手动 | 首版不做拉股价/脚本 |

## 3. 文件位置与边界

| 项 | 约定 |
|----|------|
| 台账文件 | `reports/建议跟踪台账.md` |
| 看板 | `reports/重点标的看板.md` 增加「建议台账摘要」小节，链到全表 |
| 设计文档 | `docs/2026-07-20-建议跟踪台账-design.md` |
| 与 thesis | 细节假设/红线仍在 `*-thesis.md`；台账只记建议事件 |
| 与 portfolio | 仓位仍由 `/portfolio-review` → `portfolio-latest.md` |
| 与主题监控 | 不混表 |

原则：台账不复制长文；来源用相对路径链回报告。

---

## 4. 收录范围

满足任一即可写入台账：

| 来源 | 收录规则 |
|------|----------|
| `/investment-team` 最终报告 | 有明确分层建议（买/观望/回避）或价格区间 |
| `/investment-research` | 同上 |
| `/investment-checklist` | 结论为通过/未通过且带价格含义时收录 |
| 仅有分析、无动作建议 | **不收录** |

同一公司可有多条历史建议——**按建议事件记账，不按公司去重**。

## 5. 数据模型

### 5.1 主表字段

| 字段 | 说明 | 示例 |
|------|------|------|
| ID | `YYYYMMDD-公司简称` | `20260719-合盛硅业` |
| 标的 | 公司 + 代码 | 合盛硅业 603260.SH |
| 建议日 | 报告数据截止日 | 2026-07-19 |
| 来源 | 相对路径 Markdown 链接 | `[合盛硅业/最终报告.md](合盛硅业/最终报告.md)` |
| 动作 | `买入` / `观望` / `回避` / `分批`（默认取稳健型；激进/保守可写在触发条件或复盘） | 观望 |
| 锚定价 | 建议当时的参考价 | 34.41 元 |
| 触发条件 | 可验证的一句话 | 扣非连续转正且 ≤28 元 |
| 复检日 | 到期或下次必检日 | 2026-08-31 或「下季报后」 |
| 状态 | 见 5.2 | 待触发 |
| 结果 | 见 5.3；未到期填 `—` | — |
| 复盘一句 | 事后对错与原因；未复盘填 `—` | — |
| 升级 | `无` / `应升格` / `已升格` | 无 |

### 5.2 状态（监控）

| 状态 | 含义 |
|------|------|
| `待触发` | 条件未达成，继续盯 |
| `已触发待决策` | 价格/事件条件满足，等人动作 |
| `已执行` | 已按建议买卖或明确放弃该触发 |
| `已过期` | 过了复检日仍未处理 |
| `已关闭` | 复盘完成，不再盯这条 |

### 5.3 结果（复盘；关闭时必填）

| 结果 | 含义 |
|------|------|
| `命中` | 方向与条件逻辑被事后验证为对 |
| `未命中` | 事后证明建议错了 |
| `条件未达` | 建议本身未错，但触发条件一直没出现 |
| `无法判定` | 数据不足或建议不可证伪 |

**命中率** = `命中 / (命中 + 未命中)`。  
`条件未达`、`无法判定` 单独计数，不计入分母。

### 5.4 升级门（→ `/thesis-tracker`）

满足任一，升级列标 `应升格`（或完成后 `已升格`）：

- 实际买入 / 准备纳入核心观察池
- 同一标的累计 ≥2 条未关闭建议且仍在跟踪
- 状态为 `已触发待决策` 且准备认真决策

未升格的只维护台账一行，不强制建 thesis。

## 6. 维护流程

### 6.1 产生建议时（强制）

终稿带买卖建议后：

1. 台账追加一行（§5 字段）
2. `复检日` 必填
3. 若已决定买入/进核心池 → 建 thesis，升级列写 `已升格`

**没有台账行 = 该建议未进入跟踪体系。**

### 6.2 例行节奏

| 频率 | 动作 | 预计 |
|------|------|------|
| 每周 | 扫 `待触发` / `已过期` / `已触发待决策`；更新状态；刷新看板摘要 | 15–20 分钟 |
| 触发时 | 价格进区间、事件兑现、或 `/news-pulse` 指向该标的 → 改状态；必要时升格 | 按需 |
| 关闭时 | 填结果 + 复盘一句；状态 → `已关闭` | 约 2 分钟/条 |
| 每季 | 更新命中率与条件未达计数 | 约 10 分钟 |

首版不自动拉股价；是否触发靠周扫或研究时顺手更新。

### 6.3 Skill 关系

| Skill | 关系 |
|-------|------|
| `/investment-team` 等 | 产出后登记台账一行（流程约定，首版手写） |
| `/thesis-tracker` | 仅升格标的 |
| `/news-pulse` | 异动后若命中某条条件 → 改台账状态 |
| `/portfolio-review` | 仓位另管；台账不写仓位 |
| 新 skill（如 call-ledger） | 首版不做；字段稳定后再考虑 |

## 7. 看板衔接

在 `reports/重点标的看板.md` 增加固定小节（主表仍只收录 thesis/指定标的）：

```markdown
## 建议台账摘要（链到全表）

| 指标 | 数值 |
|------|------|
| 待触发 | N |
| 已触发待决策 | N |
| 已过期未关闭 | N |
| 近季命中率 | x/y（不含条件未达） |

详情 → [建议跟踪台账](建议跟踪台账.md)
```

## 8. 首版落地步骤

1. 本设计文档定稿（本文）
2. 新建 `reports/建议跟踪台账.md`（页眉说明 + 空/初填主表 + 命中率页脚）
3. 从现有最终报告/明确建议的 research **回填**（范围由用户指定：近 N 个月 / 全量最终报告 / 名单）
4. 看板增加「建议台账摘要」
5. （可选）`CLAUDE.md` 报告命名表增加台账一行

## 9. 非目标与后续可选

**非目标（首版不做）**

- 自动验证条件是否达成
- 替代 thesis / portfolio
- 复杂命中率仪表盘
- 每公司独立 `call-card.md`（方案 3）

**后续可选（未承诺）**

- 脚本：列出「有最终建议但不在台账」的漏网公司
- `/call-ledger` skill 协助登记与周扫
- 字段稳定后拆公司卡片 + 自动汇总

使用中发现问题再改字段/枚举，不追求首版完美。

---

## 10. 验收标准

- 打开台账能看到每条建议的动作、锚定价、触发条件、状态、结果
- 看板摘要数字与台账未关闭异常项一致，并能跳到全表
- 命中率计算不含 `条件未达` / `无法判定`
- 升格标的在 thesis 看板有对应行；未升格不强制 thesis
- 台账正文无大段估值复制
